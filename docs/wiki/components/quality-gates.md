# Quality Gates

Every tailored application passes through three gates before it can move to `ready`. All three run inside `runQualityGates` in `src/core/quality/gates.ts`. A single gate failure routes the application to `quality_review` instead of `ready`. Results are stored in `applications.quality_gates_json`.

See [decisions](../decisions.md) ADR-005 for the rationale behind this design.

---

## Gate 1 — Numerics check

**File:** `src/core/quality/numerics.ts`  
**Type:** Deterministic — no LLM.

```ts
checkNumerics(bullet: string, allowed: string[]): NumericsResult
// { pass: boolean; reason?: string }
```

**Logic:**

1. Regex `/\d[\d,.\-]*/g` finds every digit run in the tailored bullet.
2. Each match is normalised (trailing punctuation stripped).
3. The normalised value must appear in `source_numbers[]` — the `numbers` allow-list from the source `ResumeBullet`.
4. First mismatch → `{ pass: false, reason: "Bullet contains digit run '...' not in source numbers[]" }`.

`checkNumericsForAll` wraps the per-bullet check across all bullet pairs, returning on the first failure.

**Why this gate:** Numeric hallucination is the most damaging class of fabrication. Claiming "led a team of 50" when the source bullet says "led a team of 12" is unverifiable during submission and could end an application or career.

## Gate 2 — Claim-equivalence judge

**File:** `src/core/quality/claim-equivalence.ts`  
**Type:** Haiku LLM call per bullet pair.

```ts
judgeClaimEquivalence({ original, tailored }): Promise<ClaimEquivResult>
// { equivalent: boolean; divergence_note: string | null }
```

**System prompt** instructs Haiku to check whether the tailored bullet introduces **any** claim not present in the original — scope, technology, team size, ownership, time period, or results. The prompt is strict: *"soft drift counts as divergence."*

`judgeAllPairs` iterates pairs and short-circuits on the first failure, returning `{ pass: false, failure: { ...result, pair } }`.

**v1 note:** Because bullets are not rewritten in v1 (tailored == original), this gate trivially passes for every application until bullet rewriting is added. The infrastructure is in place; the gate will start catching real drift once rewriting lands. See [tailoring](tailoring.md) for context.

## Gate 3 — Verbatim phrase

**Phrase source:** `src/core/tailor/verbatim-phrase.ts` — fetches the company's apply URL, strips HTML, asks Haiku to select a 5–12-word exact substring.

**Gate check:** `src/core/quality/gates.ts:35` — a plain `cover_letter.includes(verbatim_phrase)` substring test.

```ts
result.verbatim_phrase = cover_letter.includes(verbatim_phrase) ? "pass" : "fail";
```

If `verbatim_phrase` is `null` (fetch failed or no suitable phrase found), the gate auto-fails with note `verbatim_phrase: no company artifact`.

**Why this gate:** A cover letter that contains no text from the company's own materials is an AI fingerprint. Forcing an exact verbatim inclusion ensures at least one sentence is personalised and would not plausibly be generated by a prompt with no company context.

## Orchestrator

`runQualityGates(input: GatesInput): Promise<QualityGates>` in `src/core/quality/gates.ts`.

Input:
```ts
interface GatesInput {
  tailored_bullets: Array<{ tailored: string; original: string; source_numbers: string[] }>;
  cover_letter: string;
  verbatim_phrase: string | null;
}
```

Output (`QualityGates` from `src/core/applications/types.ts`):
```ts
{
  numerics: 'pass' | 'fail' | null;
  claim_equiv: 'pass' | 'fail' | null;
  verbatim_phrase: 'pass' | 'fail' | null;
  notes?: string;
}
```

The `notes` field accumulates failure reasons from each gate, pipe-separated. Stored as JSON in `applications.quality_gates_json`.

`allGatesPass(gates: QualityGates): boolean` — convenience helper used by the tailor routine to decide routing.

---

## Failure handling

| `settings.quality_review_failure_mode` | Behaviour on any gate failure |
|---|---|
| `'review'` (default) | Route to `quality_review` tray — user decides |
| `'auto_skip'` | Silently route to `ready` ignoring the failure |

The `'auto_skip'` mode exists for when the tray becomes noisy and the user trusts the current gate pass rate. It can be toggled in `/settings`.

## Tests

`tests/quality/` contains independent tests for each gate:

- Numerics: deterministic — tests cover digit-run extraction, normalisation, allow-list membership.
- Claim-equivalence: Haiku mock — tests cover the pass/fail branching and `judgeAllPairs` short-circuit.
- Verbatim phrase: substring check — tests cover present/absent/null phrase cases.
